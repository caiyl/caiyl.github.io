#java线程池的使用

##java线程池的种类
主要有4种，可以通过调研Executors种的静态工厂方法之一来创建。
newFixedThreadPool,FixedThreadPool固定线程最大数量。
newCachedThreadPool创建一个可缓存的线程池，当线程池的当前规模超过了处理请求则回收空闲的线程，该线程池的规模不存在限制。
newSingleThreadExecutor是一个单线程的Executor，它创建单个线程来执行任务，线程串行执行。
newScheduledThreadPool创建的是一个固定长度的线程池，可延迟、或定时的方法来执行任务。

##Executor的生命周期
为了解决执行服务的生命周期，Executor扩展了ExecutorService接口，添加了一些生命周期的管理方法

  1. shutdown
  2. shutdownNow
  3. isShutdown
  4. isTerminated
  5. awaitTermination

ExecutorService的生命周期有3中状态：运行、关闭、已终止。shutdown方法执行平缓的关闭过程：不再接受新的任务，同时等待已经提交的任务执行完成–包括哪些还没开始的任务。shutdownNow方法讲执行粗暴的关闭方式：它将尝试取消所有运行中的任务，并且不再启动队列中未执行的任务。

##延迟任务与周期任务
Timer类可以延迟任务已经周期任务，但是存在一些缺陷，Timer只创建一个线程。如果某个任务执行的时间过长，那么将破坏其他TimerTask的定时精确性。Timer的另一个问题的如果TimerTask跑出一个未检查的异常，那么Timer将终止定时线程，新的任务无法执行（线程泄露）。

netty的组件

发表于 2017-12-21

##netty组件
netty有8个组件组成，如下图

bootstrap与ServerBootstrap

Bootstrap用于客户端，ServerBootstrap用于服务端

Bootstrap用1个EventLoopGroup处理io事件，而ServerBootstrap用2个EventLoopGroup处理io。

ServerBootStrap中，其中一个EventLoopGroup用来处理监听接收客户端的连接，而另一个EventLoopGroup用来处理已经连接上的客户端连接。可以用一个EventLoopGroup充当两个角色即用于接收客户端连接，同时处理客户端的连接，但是当在高并发的时候，由于需要处理客户端连接，有可能导致客户端连接超时。所以在高并发的场景，最好使用两个EventLoopGroup。

###ChannelPipeline
ChannelPipeline用于添加ChannelHandler，在同一个ChannelPipeline里，通过ChannelInboundHandler 和ChannelOutboundHandler，可以在处理输入与输出。ChannelHandler中持有一个ChannelHandlerContext对象，可以通过该引用对象发送消息，也可以直接通过channel发送消息。

ChannelHandler

ChannelHandler包括ChannelInboundHandler和ChannelOutboundHandler，还有编码器StringEncoder，SimpleChannelInboundHandler等。

###EventLoopGroup与EventLoop
EventLoop的继承关系

EventLoopGroup里面包含一组EventLoop，Eventloop类继承了EventLoopGroup,当一个Channel连接的时候，从EventloopGroup中取出一个EventLoop来处理该Channel的所有io事件，所以netty不用担心线程的并发问题，因为每个channel都交由一个确定的Eventloop处理，不会改变。

冒泡排序.md

发表于 2017-11-07

冒泡排序

  1. 比较相邻的元素。如果第一个比第二个大，就交换他们两个。
  2. 对每一对相邻元素作同样的工作，从开始第一对到结尾的最后一对。这 步做完后，最后的元素会是最大的数。
  3. 针对所有的元素重复以上的步骤，除了最后一个。

  4. 持续每次对越来越少的元素重复上面的步骤，直到没有任何一对数字需要比较。

插入排序算法.md

发表于 2017-11-06

插入排序

  1. 从第一个元素开始，该元素可以认为已经被排序
  2. 取出下一个元素，在已经排序的元素序列中从后向前扫描
  3. 如果该元素（已排序）大于新元素，将该元素移到下一位置
  4. 重复步骤3，直到找到已排序的元素小于或者等于新元素的位置
  5. 将新元素插入到该位置后
  6. 重复步骤2~5

选择排序算法.md

发表于 2017-11-06

选择排序

选择排序（Selection sort）是一种简单直观的排序算法。它的工作原理如下。首先在未排序序列中找到最小（大）元素，存放到排序序列的起始位置，然后，再从剩余未排序元素中继续寻找最小（大）元素，然后放到已排序序列的末尾。以此类推，直到所有元素均排序完毕。
选择排序的主要优点与数据移动有关。如果某个元素位于正确的最终位置上，则它不会被移动。选择排序每次交换一对元素，它们当中至少有一个将被移到其最终位置上，因此对 {\displaystyle n} n个元素的表进行排序总共进行至多 {\displaystyle n-1} {\displaystyle n-1}次交换。在所有的完全依靠交换去移动元素的排序方法中，选择排序属于非常好的一种。维基百科

jvm垃圾收集器与内存分配策略.md

发表于 2017-11-05

对象已死

在回收对象之前需要确认对象是否已经死了，可以回收了。

引用计算法

引用计数法是指对对象添加一个引用的时候，引用计数器就加1，当对对象的引用少了一个的时候，计数器就减1，当对象的引用为0的时候，对象就认为已经死去。但是当对象之间互相循环引用的时候，引用计算法就无效了，例如： ObjectA a 内部持有 ObjectB b的引用，ObjectB b内部也持有ObjectA a的引用。当a=null ;b=null是，对象内部还互相引用着对方，所以无法回收对象。幸好java不是采用这种算法

根搜索算法

根搜索算法，从GC Root对象出发，搜索对象，当有引用链，让对象能链接到GC Root的时候，说明该对象还可到达，不能回收，否则回收该对象。java就是用根搜索算法回收对象。

GcRoot对象有以下这些

  • 虚拟机栈中引用的对象
  • 方法区中的类静态属性引用的对象
  • 方法区中的常量引用的对象
  • 本地方法栈中JNI的引用的对象

垃圾收集算法

标记清除算法

标记清除算法首先标记出可以回收的对象，然后一次过回收对象

复制算法

复制算法把堆内存分成两个区域，每次使用其中一块区域，当回收的时候，一次过把存活的对象拷贝到另外一块区域，然后清除该块区域。在hotSpot中，因为大部分对象都是朝生夕死，所以把堆内存分成Eden区和两块Survivor区域，每次只使用一块Eden和一块Survivor区域。Eden和Survivor的默认比例是8：1。

标记整理算法

把所有存活的对象都推向一端，然后清理边界以外的区域。

jvm的内存区域划分.md

发表于 2017-11-05

jvm运行时内存区域的划分

  • jvm虚拟机栈

    虚拟机栈是线程独享的，每个线程有自己的虚拟机栈，每个方法执行的时候创建一个栈桢，方法结束栈桢出栈。栈桢包含了局部变量表又称本地变量表，用于存放在编译期间确定的基本数据类型与对象的引用。局部变量表的大小在编译的期间已经确定，在方法运行的过程中不会改变。

  • 堆

    堆主要用来存放对象，每个线程共享堆

  • 程序计数器

    程序计数器是存在每个线程的当前执行到那条指令，用于线程要执行那条指令的时候，从中取。

  • 本地方法栈

    本地方法栈与虚拟机栈类似，本地方法栈是执行native method方法的栈

  • 方法区

    方法区又称永久代，用于存放类加载的信息，元数据，常量池

设置虚拟主机通过设置多个虚拟主机可以支持一台物理服务器访问多个域名，就好像有多个服务器一样。打开配置文件/root/mydata/apache/conf/httpd.conf，去掉#Include conf/extra/httpd-vhosts.conf前面的#，保存并退出。然后打开httpd-vhosts.conf，创建/root/mydata/esdatadir目录和/root/mydata/esdatadir/log，作为代理服务器的根路径，注释掉文件httpd-vhosts.conf中的以下代码：

     ServerName dummy-host.example.com

设置反向代理

先说一下本文档反向代理的环境，apache代理服务器安装在虚拟机上，ip地址是192.168.47.133，物理机的ip地址是192.168.2.40。为了测试反向代理，我在物理机上开了一个web应用服务，通过apche作为反向代理来访问该服务，而不是直接访问。直接访问如下图

打开配置文件httpd.conf，去掉以下module前面的#

#LoadModule proxy_module libexec/apache2/mod_proxy.so
 #LoadModule proxy_connect_module libexec/apache2/mod_proxy_connect.so
 #LoadModule proxy_ftp_module libexec/apache2/mod_proxy_ftp.so
#LoadModule proxy_http_module libexec/apache2/mod_proxy_http.so

通过以上module的命名我们可以了解到mod_proxy.so是基础的代理配置，mod_proxy_http.so支持HTTP请求，mod_proxy_ftp.so支持FTP请求，mod_proxy_connect.so 支持HTTPS请求（HTTPS请求头和报文是加密的，代理服务器不能通过识别请求头来获取目的服务器的地址，所以在最开始建立连接时代理服务器需要打开一条从客户端到服务器的端到端connect通道）
